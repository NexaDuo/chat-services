# Tenant Provisioning

(Semi-manual) automation to create a new tenant (client) in the NexaDuo stack.

## Production URLs (Reference)

- **Chatwoot:** [chat.nexaduo.com](https://chat.nexaduo.com)
- **Dify:** [dify.nexaduo.com](https://dify.nexaduo.com)
- **Evolution API (Manager):** `https://chat.nexaduo.com/evolution/manager`

### Quick Troubleshooting (404 on public URLs)

If `chat.nexaduo.com`, `dify.nexaduo.com`, or `coolify.nexaduo.com` are responding with 404 after deployment, execute:

```bash
./scripts/refresh-coolify-routes.sh
```

### Multi-tenancy Strategy (Future)
Routing for multiple tenants will be path-based:
- **Chatwoot:** `chat.nexaduo.com/{tenant}/`
- **Dify:** `dify.nexaduo.com/{tenant}/`

## Flow per Tenant

1. **Chatwoot** — new Account + admin user (via Platform API).
2. **Dify** — new App in Studio with model + Knowledge Base; export DSL to `dify-apps/`.
3. **Middleware** — add the new entry to `TENANT_MAP` in `.env`.
4. **Evolution API** — create WhatsApp instance (QR code) via Manager and activate integration with the new Chatwoot account.

## Prerequisites

- `jq`, `curl`, `openssl` on the host.
- **Chatwoot Platform API token**. It is *different* from the normal `api_access_token` — it needs to be created via Rails console (first time only):

```bash
docker compose exec chatwoot-rails bundle exec rails runner \
  'p PlatformApp.create!(name: "provisioning").access_token.token'
# Copy the returned token → export CHATWOOT_PLATFORM_TOKEN=...
```

Official documentation: [Chatwoot Platform API](https://developers.chatwoot.com/api-reference/platform).

## Script Usage

```bash
export CHATWOOT_URL=https://chat.nexaduo.com
export CHATWOOT_PLATFORM_TOKEN=<token>
export DIFY_URL=https://dify.nexaduo.com   # cosmetic only, script just prints

./provisioning/create-tenant.sh "Acme Corp" admin@acme.com "John Doe"
```

The script prints the created `account_id` and instructions to finalize setup in Dify (still manual — Dify's Console API is not stable enough for 100% automation).

## Edit TENANT_MAP

The `TENANT_MAP` is a single-line JSON in `.env`:

```env
TENANT_MAP={"1":{"dify_api_key":"app-xxxxxx"},"2":{"dify_api_key":"app-yyyyyy","dify_base_url":"http://dify-api:5001/v1"}}
```

### In Production:
The application's `.env` file in production is dynamically generated by the `deploy-tenant-direct.sh` script. To add a new tenant:
1. Update the mapping in GCP Secret Manager or adjust the template in the `deploy-tenant-direct.sh` script.
2. Re-execute the application layer deployment:
   ```bash
   ./scripts/deploy-tenant-direct.sh
   ```

### Local:
Simply edit the `.env` file at the root and restart the container:
```bash
docker compose restart middleware
docker compose logs -f middleware   # check "middleware: listening" + tenants=N
```

## Evolution API — New WhatsApp Instance

1. Open the Manager (e.g., `http://localhost:8080/manager` or the configured public URL).
2. Create a new instance, choose Chatwoot as integration, point to `http://chatwoot-rails:3000` (internal Docker network) + created `account_id` + Chatwoot bot user token.
3. Scan the QR code.

Alternative via API (no UI):

```bash
curl -sS -X POST http://localhost:8080/instance/create \
  -H "apikey: $EVOLUTION_AUTHENTICATION_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "instanceName": "acme-whatsapp",
    "qrcode": true,
    "chatwootAccountId": "1",
    "chatwootUrl": "http://chatwoot-rails:3000",
    "chatwootToken": "<bot user token>",
    "chatwootSignMsg": true
  }'
```

## Offboarding

1. `docker compose exec chatwoot-rails bundle exec rails runner 'Account.find(<id>).destroy!'`
2. Remove app from Dify Studio + delete corresponding YAML file in `dify-apps/`.
3. Remove `TENANT_MAP` entry, run `docker compose restart middleware`.
4. Remove instance in Evolution (`DELETE /instance/delete/{name}`).
