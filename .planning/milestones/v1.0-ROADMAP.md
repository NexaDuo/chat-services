# Milestone v1.0: NexaDuo Omnichannel AI Stack — MVP

**Status:** ✅ SHIPPED 2026-04-14
**Phases:** 1–5
**Total Plans:** 8 (GSD-tracked)

## Overview

Deployment of a production-ready, low-cost, multi-tenant AI-driven chat platform. The milestone delivered the full infrastructure stack — from GCP provisioning via Terraform, through Cloudflare edge routing with path-based multi-tenancy, to the AI service layer (Chatwoot, Dify, Evolution API, Middleware). Culminated in an automated provisioning CLI that registers new tenants end-to-end in minutes.

### Phase 3: Edge Logic & Path Routing

**Goal**: Implement Cloudflare Workers for header injection and path-based multi-tenant routing.
**Depends on**: Phase 2
**Status**: Completed 2026-04-14

Plans:
- [x] 03-01: Initialize Cloudflare Worker with Hono + HTMLRewriter for asset path rewriting
- [x] 03-02: Secure tenant resolution via Middleware with in-memory caching and production deployment

**Details:**
Built a Cloudflare Worker using the Hono framework to handle path-based routing (`/tenant/` → origin). Implemented HTMLRewriter to fix absolute asset paths dynamically. Added secure tenant resolution with Bearer auth to the Middleware API and a 10-minute in-memory cache. Deployed to production with routes for `chat.nexaduo.com` and `dify.nexaduo.com`.

Key decisions:
- Use Hono for its performance and clean middleware model
- HTMLRewriter over origin-side path stripping (no origin changes needed)
- 32-char SHARED_SECRET for Worker↔Middleware auth
- Inject `X-Tenant-ID` with resolved `accountId`, not URL slug

- [x] 04-02: GCP firewall hardening + Automated DNS via Terraform

**Details:**
Created a standalone TypeScript CLI using Commander + Zod for type-safe tenant registration. CLI inserts tenant records into PostgreSQL (with ON CONFLICT support), writes to `tenants.json` for Terraform consumption, and validates via Middleware `/resolve-tenant` API. Hardened GCP origin to accept ingress only from Cloudflare IP ranges (mitigating T-03-01). Automated CNAME DNS creation via Terraform `for_each` over `tenants.json`. E2E verification script tests path resolution, header injection, and HTML rewriting.

## Milestone Summary

### Key Decisions

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| Path-based routing over subdomain-per-tenant | Simplifies infrastructure — single domain pair | ✓ Working in production |
| Cloudflare Worker + Hono | Performance, clean middleware model | ✓ Deployed |
| HTMLRewriter for asset paths | No origin-side changes needed | ✓ Assets load correctly |
| GCP firewall → Cloudflare IP ranges | Mitigate T-03-01 origin bypass | ✓ Hardened |
| Terraform for_each over tenants.json | IaC-driven DNS automation | ✓ Working |
| Provisioning CLI in TypeScript | Type safety, standalone from runtime | ✓ Functional |

### Issues Resolved

- T-03-01: Cloudflare Worker origin bypass via header spoofing → Mitigated with GCP firewall IP whitelist
- WebSocket proxy compatibility through Cloudflare Worker → Fixed
- Postgres role password mismatch between .env and docker compose → Resynced (quick task)
- Asset 404s due to absolute paths in Chatwoot → Fixed via HTMLRewriter

### Issues Deferred (Tech Debt)

- **INFRA-06**: Coolify v4 application/service configuration via Terraform/API — never automated; manual Coolify UI used instead
- **03-01-SUMMARY.md**: Phase 3 Plan 01 (Worker init) has no summary — executed but not documented in GSD
- **Phases 1-2 summaries**: Pre-GSD phases have no SUMMARY.md files
- **Multitenancy at scale**: Single-node architecture not load-tested beyond initial PoC

### Technical Debt Incurred

- Worker tenant resolution cache is in-memory (resets on worker restart); no persistent cache layer
- `tenants.json` as shared state between CLI and Terraform can cause drift if modified manually
- Observability (Grafana/Prometheus) deployed but not configured with alerts or dashboards
