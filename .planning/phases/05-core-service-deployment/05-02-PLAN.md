<objective>
Add the Chatwoot Coolify stack to Terraform: a separate `coolify_service` resource pointing at `deploy/docker-compose.chatwoot.yml`, all Chatwoot-specific env vars injected via `coolify_service_envs`, depends_on the shared stack, and a post-deploy health probe over SSH that fails the `terraform apply` if Chatwoot does not become healthy. Wires `CHATWOOT_FRONTEND_URL` to the path-based multi-tenancy domain (DEPLOY-02).

Purpose: D-02 mandates a separate stack per service so Chatwoot can restart/upgrade without touching Dify. The Cloudflare Worker (Phase 3 complete) routes `chat.nexaduo.com/{tenant}/` — Chatwoot only needs `FRONTEND_URL` to match so it generates correct asset and redirect URLs.

Output: Idempotent Terraform that deploys nexaduo-chatwoot-init, nexaduo-chatwoot-rails, and nexaduo-chatwoot-sidekiq via Coolify in its own stack, sharing `nexaduo-network` with the shared stack.
</objective>

    ```hcl
    # ---------------------------------------------------------------------------
    # Stack 2/4 — Chatwoot (init + rails + sidekiq). Joins nexaduo-network so
    # Postgres + Redis (shared stack) are reachable by container name.
    # ---------------------------------------------------------------------------
    resource "coolify_service" "chatwoot" {

      server_uuid      = data.coolify_servers.main.servers[0].uuid

      depends_on = [
        coolify_service.shared,
        coolify_service_envs.shared,
        null_resource.verify_shared,
      ]
    }

    resource "coolify_service_envs" "chatwoot" {
      uuid = coolify_service.chatwoot.uuid

      # Shared infra references (must match the values injected into the shared stack)
      env {
        key   = "POSTGRES_HOST"
        value = "postgres"
      }
      env {
        key   = "POSTGRES_PORT"
        value = "5432"

        value      = var.redis_password
        is_literal = true
      }
      env {
        key   = "TZ"
        value = var.tz
      }

      # Chatwoot-specific
      env {
        key        = "CHATWOOT_SECRET_KEY_BASE"
        value      = var.chatwoot_secret_key_base
        is_literal = true
      }
      env {
        key   = "CHATWOOT_FRONTEND_URL"
        value = var.chatwoot_frontend_url
      }
      env {

        value = "NexaDuo"

      env {

        value = "pt_BR"

      env {
        key   = "CHATWOOT_ENABLE_ACCOUNT_SIGNUP"
        value = "false"
      }
      env {

        value = "false"
      }
    }

    # ---------------------------------------------------------------------------
    # Post-deploy health probe (D-05): wait for chatwoot-rails to be healthy AND
    # serve HTTP 200 on / via the host-published port 3000.
    # ---------------------------------------------------------------------------
    resource "null_resource" "verify_chatwoot" {
      depends_on = [coolify_service.chatwoot, coolify_service_envs.chatwoot]

      connection {
        type        = "ssh"
        user        = var.ssh_user
        private_key = file(var.ssh_private_key_path)
        host        = module.vm.public_ip
      }

      provisioner "remote-exec" {
        inline = [
          "echo 'Waiting for nexaduo-chatwoot-rails to be healthy (up to 6 min)'",
          "timeout 360 bash -c 'until [ \"$(docker inspect -f \"{{.State.Health.Status}}\" nexaduo-chatwoot-rails 2>/dev/null)\" = \"healthy\" ]; do sleep 5; done'",
          "echo 'Probing http://localhost:3000/ for HTTP 200 (up to 1 min)'",
          "timeout 60 bash -c 'until [ \"$(curl -s -o /dev/null -w %%{http_code} http://localhost:3000/)\" = \"200\" ]; do sleep 3; done'",
          "echo 'OK chatwoot stack healthy'"
        ]
      }

      triggers = {
        compose_hash = filesha256("${path.root}/../../../deploy/docker-compose.chatwoot.yml")
      }
    }

    output "coolify_chatwoot_service_uuid" {
      value = coolify_service.chatwoot.uuid
    }
    ```

    Why each env var here even though chatwoot.yml has `env_file: ../.env`:
    - `coolify_service_envs` is the Terraform-managed source of truth (D-03). Coolify writes these into the deployed stack environment so compose `${...}` interpolations resolve. The compose `env_file: ../.env` line stays as a no-op safety net; Coolify-managed envs take precedence at runtime.
    - `is_literal = true` on secrets prevents Coolify from interpreting `$`-prefixed substrings inside the value as Coolify magic vars.
    - `CHATWOOT_FRONTEND_URL = var.chatwoot_frontend_url` (default `https://chat.nexaduo.com`) — satisfies DEPLOY-02. The Cloudflare Worker handles `/{tenant}/` path stripping at the edge; Chatwoot itself only needs the public origin URL.
    - The health probe also curl-checks port 3000 directly — `depends_on` only sequences the Terraform graph; the Docker healthcheck status + a real HTTP probe is what proves the app is actually serving requests.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Local dev / CI runner -> Coolify API | Bearer-token requests to add the chatwoot service + env vars |
| Local dev / CI runner -> GCP VM (SSH) | `null_resource.verify_chatwoot` opens SSH for the health probe |
| Cloudflare Worker -> Chatwoot origin (port 3000) | Public ingress flows through the Worker; Chatwoot trusts the path-stripped request |
| Chatwoot container -> Postgres / Redis (shared stack) | Cross-stack via the shared `nexaduo-network` bridge |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-07 | Tampering | `CHATWOOT_SECRET_KEY_BASE` value passing through Coolify | mitigate | `is_literal = true` on the env block prevents Coolify from interpreting `$`-prefixed sequences as magic vars (Task 1). Variable already declared `sensitive = true` (05-01). |
| T-05-08 | Information Disclosure | Chatwoot port 3000 published on the host | accept | The host firewall (Phase 4) restricts ingress to Cloudflare IPs; only the Worker can reach 3000. Documented and accepted in RESEARCH.md Security Domain. |
| T-05-09 | Spoofing | Cross-stack name resolution (chatwoot -> postgres on nexaduo-network) | mitigate | Network is private to the host; no external attachments. The fixed network name (05-01 Task 2) prevents accidental UUID-based collisions but does not introduce new ingress. |
| T-05-10 | Repudiation | Chatwoot logs sent to Loki (configured in 05-04) | mitigate | Promtail (deployed in nexaduo stack, plan 05-04) tails Docker container stdout — Chatwoot logs are captured automatically once Loki is up. |
</threat_model>
