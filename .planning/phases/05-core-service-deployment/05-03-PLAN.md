<objective>
Add the Dify Coolify stack to Terraform: separate `coolify_service` resource pointing at `deploy/docker-compose.dify.yml`, all Dify env vars injected via `coolify_service_envs`, depends_on the shared stack, and a post-deploy health probe over SSH that fails the `terraform apply` if the Dify API does not become responsive. Wires `DIFY_CONSOLE_API_URL` and `DIFY_APP_API_URL` to the path-based multi-tenancy domain (DEPLOY-02).

Purpose: D-02 separates Dify's lifecycle from Chatwoot. The Cloudflare Worker (Phase 3 complete) routes `dify.nexaduo.com/{tenant}/` — Dify needs its console/app URL envs to match so generated links and CORS responses are correct.

Output: Idempotent Terraform that deploys all 6 Dify containers (dify-init, dify-api, dify-worker, dify-web, dify-sandbox, dify-plugin-daemon, dify-ssrf-proxy) via Coolify in its own stack, sharing `nexaduo-network` with the shared stack.
</objective>

    # Stack 3/4 — Dify (api + worker + web + sandbox + plugin-daemon + ssrf-proxy).

      name             = "nexaduo-dify"
      server_uuid      = data.coolify_servers.main.servers[0].uuid
      project_uuid     = coolify_project.main.uuid
      environment_name = "production"

      compose = file("${path.root}/../../../deploy/docker-compose.dify.yml")

      depends_on = [
        coolify_service.shared,
        coolify_service_envs.shared,
        null_resource.verify_shared,
      ]
    }

    resource "coolify_service_envs" "dify" {
      uuid = coolify_service.dify.uuid

      # Shared infra references
      env {

        value = "postgres"

      env {

      }
      env {

      }
      env {
        key        = "REDIS_PASSWORD"

      env {

        is_literal = true

        is_literal = true

      env {
        key   = "DIFY_API_ENABLE_METRICS"
        value = "true"
      }
      env {
        key   = "INNER_API_METRICS_ENABLED"
        value = "true"
      }
    }

    # ---------------------------------------------------------------------------
    # Post-deploy health probe (D-05): wait for nexaduo-dify-api healthy AND
    # serve HTTP 200 on /console/api/setup via host-published port 5001.
    # ---------------------------------------------------------------------------
    resource "null_resource" "verify_dify" {
      depends_on = [coolify_service.dify, coolify_service_envs.dify]

      connection {
        type        = "ssh"
        user        = var.ssh_user
        private_key = file(var.ssh_private_key_path)
        host        = module.vm.public_ip
      }

      provisioner "remote-exec" {
        inline = [
          "echo 'Waiting for nexaduo-dify-api to start (up to 5 min)'",
          "timeout 300 bash -c 'until docker inspect nexaduo-dify-api >/dev/null 2>&1; do sleep 5; done'",
          "echo 'Probing http://localhost:5001/console/api/setup for HTTP 200 (up to 3 min)'",
          "timeout 180 bash -c 'until [ \"$(curl -s -o /dev/null -w %%{http_code} http://localhost:5001/console/api/setup)\" = \"200\" ]; do sleep 5; done'",
          "echo 'OK dify stack healthy'"
        ]
      }

      triggers = {

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Local dev / CI runner -> Coolify API | Bearer-token requests to add the dify service + env vars |
| Local dev / CI runner -> GCP VM (SSH) | `null_resource.verify_dify` opens SSH for the health probe |
| Cloudflare Worker -> Dify origin (port 5001/3001) | Public ingress flows through the Worker |
| Dify API -> dify-sandbox (port 8194) | Internal Python code execution; sandbox is jailed |
| dify-ssrf-proxy -> external internet | Squid proxy for restricted outbound HTTP from Dify (workflow tool calls) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-11 | Tampering | DIFY_SECRET_KEY, DIFY_SANDBOX_API_KEY, DIFY_PLUGIN_DAEMON_KEY values | mitigate | All marked `is_literal = true` in `coolify_service_envs.dify` so Coolify does not interpret `$`-prefixed substrings as magic vars (Task 1). Variables already declared `sensitive = true` in 05-01. |
| T-05-12 | Information Disclosure | `*_CORS_ALLOW_ORIGINS = "*"` (set inside the compose file directly, not via Terraform) | accept | Wide CORS is required by the path-routed multi-tenant model where each tenant has a distinct origin path. Cloudflare Worker is the actual gatekeeper at the edge. Documented and accepted in RESEARCH.md Security Domain. |
| T-05-13 | Elevation of Privilege | dify-sandbox executing user-provided Python | mitigate | Sandbox container runs in the isolated `langgenius/dify-sandbox:0.2.14` image with its own seccomp profile (provided by the image). No mitigation added in this plan; relies on upstream sandbox guarantees. |
| T-05-14 | Spoofing | dify-plugin-daemon validating signatures from the Dify Marketplace | accept | `FORCE_VERIFYING_SIGNATURE` is left at `false` in the compose file (matches dev convention). Operator must flip to `true` for production marketplace use — out of scope for this plan; documented as known caveat. |
</threat_model>
