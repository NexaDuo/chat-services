# Quick Task 260414-l4s: Run docker compose up -d and fix console errors

## Context

Running `docker compose up -d` surfaced repeated `password authentication failed for user "postgres"` errors across multiple dependent services (chatwoot-init, dify-api, dify-plugin-daemon, evolution-api, self-healing-agent). Postgres itself reported healthy.

Root cause: the `postgres` role password inside the existing `postgres-data` volume did not match `POSTGRES_PASSWORD` in `.env`. Docker's official Postgres images only honor `POSTGRES_PASSWORD` at initial volume creation — rotating the `.env` value after a first-boot leaves the DB role holding the previous secret. Local `psql` connections worked because of `trust` auth on the unix socket, which masked the drift.

## Tasks

### Task 1: Diagnose failing services

- files: (runtime only)
- action: Run `docker compose up -d`, then inspect `docker compose ps --all` and `docker logs <svc>` for each restarting/exited container to classify errors.
- verify: A single shared root cause identified for all failures.
- done: Root cause = stale `postgres` role password inside the volume vs. `.env`.

### Task 2: Reset postgres role password to match .env

- files: (runtime only — no source changes)
- action: `ALTER USER postgres WITH PASSWORD '<POSTGRES_PASSWORD from .env>'` via `docker exec nexaduo-postgres psql`. Validate by connecting from a sibling container on `nexaduo-network` using the new password.
- verify: `psql -h postgres -U postgres -d chatwoot -c 'SELECT 1'` returns `1` from another container.
- done: Password auth succeeds over the compose network.

### Task 3: Restart dependent services and confirm clean state

- files: (runtime only)
- action: `docker compose restart chatwoot-init chatwoot-rails chatwoot-sidekiq dify-api dify-plugin-daemon evolution-api self-healing-agent`, then `docker compose up -d` again to verify idempotence.
- verify: `docker compose ps --all` shows every long-running service `Up` (healthy where applicable); init containers exit with code 0.
- done: No container in Restarting state; no auth errors in logs.
