# Quick Task 260414-l9x: Run docker compose up -d on clean env and fix console errors

## Context

Immediately after `docker compose down -v` (previous quick task 260414-l4s removed all named volumes), bring the stack back up with `docker compose up -d` and confirm no regressions surface on a fresh initialization.

With volumes wiped, `POSTGRES_PASSWORD` from `.env` is used to initialize the `postgres` role for the first time, so the drift issue that motivated 260414-l4s cannot recur on this run.

## Tasks

### Task 1: Launch the stack

- files: (runtime only)
- action: `docker compose up -d`.
- verify: Command exits 0, no `didn't complete successfully` messages on stdout.
- done: Orchestrated startup finishes without a compose-level failure.

### Task 2: Confirm steady state

- files: (runtime only)
- action: Inspect `docker compose ps --all`; scan `docker logs` for `dify-api`, `evolution-api`, `chatwoot-init` for error patterns; monitor for ~2 minutes to rule out restart loops.
- verify: All long-running services Up (healthy where a healthcheck is defined); `chatwoot-init` and `dify-init` exit 0; no `Restarting` states; no auth/migration errors in logs.
- done: Zero failing services; zero error-level log lines needing action.
